## Быстрый Конспект

> [!summary] Запомнить
> Git - это контроль над изменениями. Для DevOps он важен не меньше, чем для разработчика, потому что через Git часто меняются конфиги, pipelines, deploy scripts и инфраструктура.

**Главное:**

- Личный цикл: `status` -> `add` -> `commit` -> `push`.
- Командный цикл: `clone` -> `новая branch` -> `commits` -> `push` -> `Pull Request` -> `review` -> `merge`
- `main/master` держим стабильной, для задач создаём отдельные branches.
- Перед изменением DevOps-файлов думать о последствиях для deploy/production.

**Практика после видео:** создать ветку, сделать commit, запушить, открыть Pull Request и посмотреть diff.

## Идеи Без Воды

> [!ai] AI extraction
> Главные мысли, которые реально стоит оставить в Obsidian.

- Git нужен не только программистам. Для DevOps это базовый рабочий инструмент: через него меняют конфиги, CI/CD pipelines, deploy scripts, infrastructure as code и документацию.
- Git решает две главные задачи: хранит историю изменений и позволяет безопасно работать в отдельных ветках, не ломая основную версию.
- Git работает локально на компьютере, а GitHub/GitLab/Bitbucket нужны для совместной работы и хранения удалённой копии репозитория.
- Основной личный цикл Git простой: `git status` -> `git add` -> `git commit` -> `git push`.
- В командной работе нельзя просто пушить куда хочется. Обычно работа идёт через отдельную branch и Pull Request / Merge Request.
- `main` или `master` обычно считается стабильной веткой. В компаниях туда часто нельзя коммитить напрямую.
- Feature branch нужна, чтобы делать одну задачу отдельно от основной разработки.
- Pull Request нужен не для бюрократии, а для review, обсуждения, проверки и защиты стабильной версии проекта.
- `merge` сохраняет историю слияния веток, а `rebase` делает историю более линейной, но переписывает её.
- `rebase` нельзя делать на общей ветке, которой уже пользуются другие люди, иначе можно сломать им историю.
- `cherry-pick` полезен, когда нужно перенести один конкретный commit в другую ветку, но злоупотреблять им нельзя из-за риска дублирования и конфликтов.
- В DevOps иногда изменения делаются быстрее и ближе к production, поэтому важны не только команды Git, но и понимание последствий каждого изменения.
- Главный вывод: Git даёт уверенность, что изменения контролируемы, историю можно посмотреть, а ошибку можно найти или откатить.

## Что Применить На Практике

- [ ] Каждый раз перед работой выполнять `git status`, чтобы понимать, где находишься и что изменено.
- [ ] Перед `push` проверять, что изменения действительно сохранены в файлах и попали в commit.
- [ ] Для каждой учебной задачи создавать отдельную ветку: `git checkout -b practice/topic-name`.
- [ ] После изменения файла делать полный цикл: `git add .`, `git commit -m "Clear message"`, `git push`.
- [ ] На GitHub потренироваться создать Pull Request из своей branch в `main`.
- [ ] Потренироваться читать историю: `git log --oneline`, `git diff`, `git show <commit>`.
- [ ] Сделать учебный конфликт в двух ветках и попробовать решить его руками.
- [ ] Отдельно потренировать `merge` и `rebase` на тестовом репозитории, не на важном проекте.
- [ ] Никогда не делать `rebase` общей ветки, если не понимаешь, кто ещё ей пользуется.
- [ ] Для DevOps-привычки перед изменением config/pipeline задавать вопрос: “что сломается, если это сразу попадёт в production?”

## Термины

| Термин | Простое объяснение |
| --- | --- |
| Git | Локальная система контроля версий: хранит историю изменений и помогает работать с ветками. |
| GitHub | Онлайн-сервис для хранения Git-репозиториев и командной работы. |
| Repository | Папка проекта вместе с историей Git. |
| Commit | Сохранённая точка изменений с описанием. |
| Branch | Отдельная линия разработки, где можно безопасно делать задачу. |
| `main` / `master` | Главная стабильная ветка проекта. |
| Feature branch | Ветка под отдельную задачу или фичу. |
| Pull Request | Запрос на добавление изменений из одной ветки в другую на GitHub. |
| Merge Request | То же самое, но обычно в GitLab. |
| Review | Проверка изменений другим человеком. |
| Merge | Объединение веток с сохранением истории слияния. |
| Rebase | Перенос commits поверх другой ветки для более чистой линейной истории. |
| Cherry-pick | Перенос одного конкретного commit в другую ветку. |
| Conflict | Ситуация, когда Git не может сам объединить изменения. |
| Pipeline | Автоматический процесс сборки, тестов или деплоя. |
| Production | Рабочая среда, которой пользуются реальные пользователи. |

## Важное Для DevOps

> [!warning] Осторожно
> DevOps часто работает с файлами, которые влияют на деплой, окружения и production. Ошибка в Git может быть не просто “плохой commit”, а сломанный deploy или упавший сервис.

Перед изменением infrastructure/config/pipeline проверять:

1. В какой ветке я нахожусь?
2. Что именно изменилось? `git diff`
3. Есть ли быстрый способ отката?
4. Кто ещё использует эту ветку?
5. Запустятся ли pipeline/test после push?
