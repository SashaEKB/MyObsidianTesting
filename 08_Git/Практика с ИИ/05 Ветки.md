# Ветки

Ветка — отдельная линия разработки.

Главная ветка чаще всего называется `main`.

Ветка нужна, чтобы делать изменения отдельно от `main` и не ломать основную версию.

## Посмотреть ветки

```bash
git branch
```

Показывает список локальных веток.

Звездочка `*` показывает текущую ветку.

## Создать новую ветку

```bash
git branch test-branch
```

Создает ветку `test-branch`, но не переключает на нее.

## Переключиться на ветку

```bash
git switch test-branch
```

`git switch` переключает тебя на другую ветку.

## Создать и сразу перейти

```bash
git switch -c test-branch
```

`-c` значит create, то есть создать новую ветку и сразу перейти на нее.

В реальных проектах ветки часто называют по смыслу задачи:

```text
feature/test-branch
practice/git-branches
fix/gitignore
docs/git-notes
```

Пример:

```bash
git switch -c feature/test-branch
```

## Вернуться на главную ветку

```bash
git switch main
```

Возвращает тебя на ветку `main`.

## Слить ветку в main

```bash
git merge test-branch
```

`git merge` переносит изменения из одной ветки в текущую.

Перед `merge` обычно нужно быть на ветке `main`.

## Отправить новую ветку на GitHub

Если ветка новая, первый push делается так:

```bash
git push -u origin feature/test-branch
```

Смысл:

- `origin` — удаленный репозиторий на GitHub;
- `feature/test-branch` — имя ветки;
- `-u` связывает локальную ветку с веткой на GitHub.

После этого Git запоминает связь:

```text
feature/test-branch -> origin/feature/test-branch
```

И следующие разы можно писать просто:

```bash
git push
```

## Что GitHub предлагает после push новой ветки

После `git push -u origin feature/test-branch` GitHub может написать:

```text
Create a pull request for 'feature/test-branch'
```

Это не ошибка.

GitHub говорит: “Ты отправил новую ветку. Хочешь объединить её с `main` через Pull Request?”

## Pull Request

Pull Request — это запрос на слияние ветки.

Простыми словами:

```text
feature/test-branch содержит новые изменения
main пока этих изменений не содержит
Pull Request нужен, чтобы перенести изменения из feature/test-branch в main
```

На GitHub обычно видно:

```text
base: main
compare: feature/test-branch
```

Смысл:

```text
влить feature/test-branch в main
```

## Merge Pull Request

Когда нажимаешь `Merge pull request`, GitHub объединяет изменения из ветки в `main`.

Иногда GitHub создаёт отдельный merge commit:

```text
Merge pull request #1 from SashaEKB/feature/test-branch
```

Это нормально.

После merge изменения уже находятся в `main` на GitHub.

## Что делать локально после merge на GitHub

После merge на сайте нужно обновить локальную ветку `main`:

```bash
git switch main
git pull
```

`git pull` скачает изменения из GitHub в локальную `main`.

Если feature-ветка больше не нужна, её можно удалить локально:

```bash
git branch -d feature/test-branch
```

Удалить ветку на GitHub:

```bash
git push origin --delete feature/test-branch
```

## Рабочий цикл с веткой

```bash
git switch main
git pull
git switch -c feature/test-branch
```

Дальше меняешь файлы и делаешь commit:

```bash
git status
git add .
git commit -m "first feature branch"
```

Отправляешь ветку на GitHub:

```bash
git push -u origin feature/test-branch
```

На GitHub создаёшь Pull Request и делаешь merge.

Потом локально:

```bash
git switch main
git pull
git branch -d feature/test-branch
```

## Важная мысль

`git push` новой ветки не меняет `main`.

Он только отправляет отдельную ветку на GitHub.

Чтобы изменения попали в `main`, нужен один из вариантов:

- Pull Request и `Merge pull request` на GitHub;
- или локально `git switch main` -> `git merge feature/test-branch` -> `git push`.
