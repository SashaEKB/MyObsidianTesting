**Главное:**

- Сначала понять, нравится ли сам процесс DevOps/Linux, а не только зарплата.

- Собеседование - это навык, а не экзамен личности.
- Резюме должно показывать конкретные действия и стек, а не общие фразы.

## Идеи Без Воды

> [!ai] AI extraction
> Главные мысли, которые реально стоит оставить в Obsidian.

- Не выбирать профессию только из-за денег. Сначала нужно понять, нравится ли сам процесс: задачи, инструменты, стиль работы, ежедневная рутина.
- Направление лучше выбирать не по названию, а по реальным вакансиям: какие задачи, какой стек, какие требования, какая зарплата, какой рост.
- Перед глубоким обучением нужно “попробовать профессию на вкус”: посмотреть реальные задачи, поговорить с людьми из профессии, сделать маленький практический проект.
- Цель начального обучения не “знать всё”, а закрыть базу и научиться отвечать на типовые вопросы собеседований.
- Собеседование - отдельный навык. Его нужно тренировать так же, как Git, Linux или английский.
- Лучший материал для подготовки - реальные собеседования, вопросы из вакансий и повторяющиеся темы.
- После базовой подготовки не нужно бесконечно “ещё чуть-чуть учиться”. Нужно начинать получать реальную обратную связь через собеседования.
- Резюме должно быть коротким, структурным и релевантным вакансии. HR смотрит быстро, поэтому вода убивает шанс.
- В резюме важны конкретные результаты, стек и совпадение ключевых слов с вакансией.
- Спорный совет из видео про “придумать опыт” лучше заменить безопасным подходом: честно упаковать учебные проекты, практику, open-source, лабораторные задачи и показать их как опыт решения реальных задач.
- Отказы на собеседованиях - не провал, а тренировка. После 10-20 собеседований вопросы начинают повторяться.
- Первый offer не всегда лучший. Нужно уметь сравнивать условия, команду, задачи, рост, зарплату и риски.
- Зарплату можно обсуждать спокойно и профессионально, без ультиматумов.
- Испытательный срок - отдельный этап: нужно не только делать задачи, но и понимать процессы компании, задавать вопросы, быть предсказуемым и не создавать хаос.
- После трудоустройства нельзя расслабляться: рынок меняется, навыки устаревают, резюме и навык интервью нужно поддерживать.
- Для роста полезен mentor: человек, который видит ошибки быстрее тебя и помогает не тратить месяцы впустую.

## Что Применить На Практике

- [ ] Выписать 5 возможных направлений в IT/DevOps, которые потенциально интересны.
- [ ] По каждому направлению открыть 10-20 вакансий и выписать повторяющиеся требования.
- [ ] Оставить 1-2 направления, где совпадают интерес, задачи, зарплата и реальность входа.
- [ ] Найти 1-2 человека из выбранного направления и задать им вопросы о реальной работе.
- [ ] Собрать список типовых вопросов для собеседований по DevOps/Linux/Git.
- [ ] Смотреть реальные собеседования и выписывать вопросы, на которые не можешь ответить.
- [ ] Делать карточки/заметки по каждому вопросу: короткий ответ, пример, команда, частая ошибка.
- [ ] Подготовить резюме под конкретное направление, а не “обо всём”.
- [ ] В резюме писать не “учил Linux”, а “настроил Ubuntu, systemd service, Git workflow, SSH, права доступа”.
- [ ] Сделать 2-3 демонстрационных проекта, которые можно честно показывать как практику.
- [ ] После каждого учебного собеседования записывать вопросы и обновлять базу знаний.
- [ ] Вести таблицу откликов: дата, компания, вакансия, статус, вопросы, что улучшить.
- [ ] Регулярно обновлять резюме и GitHub/портфолио.
- [ ] Не принимать offer сразу эмоционально: сравнить задачи, наставничество, команду, зарплату, испытательный срок.

## Практичный План Для Тебя

> [!workflow] Цель
> Не “войти в IT любой ценой”, а постепенно стать человеком, который умеет делать понятные DevOps/Linux задачи и может это показать.

1. **База:** Linux, terminal, Git, SSH, permissions, systemd, networking basics.
2. **Практика:** маленькие проекты, где есть реальные команды и ошибки.
3. **Портфолио:** GitHub repositories с README: что сделал, как запустить, чему научился.
4. **Собеседования:** список вопросов + ответы в Obsidian.
5. **Резюме:** коротко, без воды, под конкретное направление.
6. **Английский:** параллельно учить IT English, потому что он расширяет рынок и доступ к документации.

## Как Упаковывать Опыт Честно

Вместо сомнительного “придумать опыт” использовать формулировки, которые можно защитить на вопросах:

| Слабая формулировка | Лучше |
| --- | --- |
| Учусь Linux | Настроил Ubuntu environment, разобрал права, пользователей, systemd basics. |
| Немного знаю Git | Вёл Git repository, делал commits, branches, push/pull, решал базовые ошибки. |
| Проходил курс | Сделал учебный проект и описал результат в README. |
| Нет опыта | Есть практические лабораторные задачи и documented learning projects. |

## Вопросы Для Самопроверки

- Почему тебе интересен DevOps/Linux, кроме зарплаты?
- Какие 10 команд Linux ты реально использовал и можешь объяснить?
- Что происходит при `git add`, `git commit`, `git push`?
- Как бы ты объяснил ошибку `nothing to commit, working tree clean`?
- Какой проект ты можешь показать на GitHub уже сейчас?
- Какие вопросы собеседования уже повторяются в твоих заметках?

## Что Не Брать Из Видео Буквально

> [!warning] Фильтр качества
> Видео мотивирующее, но часть советов агрессивная и спорная. Для долгой карьеры лучше строить доверие: не врать, а честно усиливать практику, проекты, объяснения и портфолио.

- Не стоит указывать навыки, которые совсем не понимаешь. Лучше быстро закрыть базу и указывать то, что можешь объяснить.
- Не стоит строить резюме на выдуманной истории. Лучше сделать учебные проекты похожими на рабочие задачи.
- Не стоит откликаться хаотично без анализа. Нужна статистика: сколько откликов, сколько ответов, где провалы.
- Не стоит воспринимать первое трудоустройство как финал. Это начало практического обучения.
